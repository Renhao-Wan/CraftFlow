# 优化 11：Editor Prompt 精简（P2）

## 现状分析

### Editor Prompt 结构

**文件**: `app/graph/polishing/debate/prompts.py` 第 74-118 行

`EDITOR_SYSTEM_PROMPT` 是独立定义的完整 Prompt（不通过 `create_base_system_prompt`），包含以下部分：

1. **角色定义**（~50 字符）
2. **核心原则**（~100 字符）：严格压分、宁严勿松、具体指出问题
3. **评分维度**（~500 字符）：4 维度 × 扣分标准
4. **评分纪律**（~150 字符）：首轮不超过 80 分、至少 3 条建议、逐条检查
5. **JSON 输出格式示例**（~400 字符）：完整的 JSON 结构 + 示例数据

**总计**：~1,200 字符 ≈ ~750 token

### 每轮 Editor 调用的完整输入

| 项目 | Token 估算 |
|------|-----------|
| System prompt（EDITOR_SYSTEM_PROMPT） | ~750 token |
| Human message（待评估文章 + 迭代轮次） | ~2000-3000 token（取决于文章长度） |
| **合计** | **~2750-3750 token** |

### 可优化的部分

#### 1. 扣分标准过于详细（~200 字符可压缩）

当前：
```
1. **逻辑性（25 分）** — 论点是否明确、论证是否充分、段落衔接是否自然
   - 扣分标准：论点模糊 -5~10, 论据不足 -5~10, 段落跳跃 -3~8

2. **可读性（25 分）** — 语言是否流畅、句式是否多样、段落长度是否合理
   - 扣分标准：表达生硬 -5~10, 句式单调 -3~8, 段落过长/过短 -3~5

3. **准确性（25 分）** — 事实是否正确、数据是否可靠、引用是否准确
   - 扣分标准：事实错误 -8~15, 数据过时 -5~10, 引用不规范 -3~8

4. **专业性（25 分）** — 内容深度、术语规范性、观点独到性
   - 扣分标准：内容浅显 -5~10, 术语不规范 -3~8, 观点平庸 -3~8
```

扣分标准的具体数值（-5~10 等）对 LLM 来说不是必须的 — LLM 能理解"严格评分"的语义，不需要精确的数值范围。

#### 2. JSON 输出示例过长（~300 字符可压缩）

当前示例包含完整的 4 维度评分数据：
```json
{
  "scores": [
    {"dimension": "逻辑性", "score": 18, "comment": "第二段与第三段之间缺少过渡，论点在第四段才明确"},
    {"dimension": "可读性", "score": 16, "comment": "前三段连续使用长句，建议拆分"},
    {"dimension": "准确性", "score": 20, "comment": "数据来源未标注，部分引用缺少出处"},
    {"dimension": "专业性", "score": 17, "comment": "技术深度不足，缺少实际案例支撑"}
  ],
  "total_score": 71,
  "feedback": "文章整体框架尚可，但存在以下关键问题需要改进...",
  "highlights": ["亮点1"],
  "improvements": ["改进1", "改进2", "改进3"]
}
```

可以精简为结构骨架 + 1 个维度示例。

## 优化方案

### 精简后的 EDITOR_SYSTEM_PROMPT

```python
EDITOR_SYSTEM_PROMPT = """你是一位极其严格的主编，用高标准审视文章质量，绝不轻易放过问题。

## 核心原则

- **严格压分**：大多数文章首轮评分应在 55-75 之间
- **宁严勿松**：宁可多挑毛病，不可敷衍打高分
- **具体指出问题**：每条反馈必须指向文章中的具体段落或句子

## 评分维度（总分 100 分，每维度 25 分）

1. **逻辑性** — 论点明确性、论证充分性、段落衔接
2. **可读性** — 语言流畅度、句式多样性、段落长度合理性
3. **准确性** — 事实正确性、数据可靠性、引用准确性
4. **专业性** — 内容深度、术语规范性、观点独到性

## 评分纪律

- 首轮总分不得超过 80 分
- 每轮至少 3 条具体改进建议
- 下轮必须逐条检查上轮改进是否落实

## 输出格式

```json
{
  "scores": [
    {"dimension": "逻辑性", "score": 18, "comment": "具体问题描述"},
    {"dimension": "可读性", "score": 16, "comment": "具体问题描述"},
    {"dimension": "准确性", "score": 20, "comment": "具体问题描述"},
    {"dimension": "专业性", "score": 17, "comment": "具体问题描述"}
  ],
  "total_score": 71,
  "feedback": "整体评价（1-2 句话）",
  "highlights": ["亮点"],
  "improvements": ["改进1", "改进2", "改进3"]
}
```"""
```

### 精简对比

| 部分 | 改前 | 改后 | 节省 |
|------|------|------|------|
| 角色定义 | ~50 字符 | ~25 字符 | ~25 字符 |
| 核心原则 | ~100 字符 | ~80 字符 | ~20 字符 |
| 评分维度 | ~500 字符（含扣分标准） | ~200 字符（去掉扣分数值） | ~300 字符 |
| 评分纪律 | ~150 字符 | ~100 字符 | ~50 字符 |
| JSON 示例 | ~400 字符 | ~250 字符（精简 comment） | ~150 字符 |
| **总计** | **~1,200 字符** | **~655 字符** | **~545 字符（~45%）** |

**Token 节省**：~750 token → ~400 token，每轮 Editor 调用节省 ~350 token。

### 注意事项

**输入不受限制**：此优化仅精简 Editor 的 **system prompt**（固定模板）。Editor 的**输入**（待评估的完整文章）可以任意长，不受此优化影响。即使文章有 10,000 字，system prompt 的精简不会影响 Editor 读取和评估完整文章的能力。

### 对评分质量的影响

精简后的 Prompt 保留了所有关键约束：
- ✅ 严格压分（首轮 ≤ 80）
- ✅ 四维度评分（各 25 分）
- ✅ 每轮 ≥ 3 条改进建议
- ✅ 逐条检查落实
- ✅ JSON 输出格式

去掉的内容：
- ❌ 扣分标准的具体数值范围（-5~10 等）— LLM 能理解"严格"的语义
- ❌ 过于详细的 JSON 示例数据 — 保留结构骨架即可

## 难度与风险

| 实现成本 | **低** — 纯文本修改，不改代码逻辑 |
| 风险 | **低** — 保留了所有关键评分约束 |
| 质量影响 | 需验证 — 精简后评分是否仍然严格 |

## 实施建议

1. 先用同一组文章对比改前/改后的 Editor 评分结果
2. 重点关注：
   - 首轮评分是否仍在 55-75 区间（而非偏高）
   - feedback 和 improvements 是否仍然具体（而非笼统）
   - 逐条检查落实的机制是否仍然有效
3. 如果评分质量下降（如偏高或笼统），恢复扣分标准部分
4. 可结合 A/B 测试：同一文章分别用精简/完整 Prompt 评分，对比结果差异
